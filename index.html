The idea

Curated Connections is a $49/mo community matchmaking SaaS — automated 1-on-1 matching, member directories, accountability challenges, group events. Built on Rails + Turbo/Stimulus. The kind of product that benefits enormously from an always-on AI handling the routine stuff.

The experiment: can an AI employee 10x the revenue of a two-person SaaS? Not by doing busywork — by doing everything. Product development, customer support, log investigation, marketing asset creation, conversion optimization, onboarding automation, content writing, A/B testing, bug fixes, feature builds. It's smarter than any individual human at every one of these tasks, and it's only getting smarter through 2026. The infrastructure is a MacBook Air M1, Claude Code on a $200/mo Max subscription, and Slack as the interface. The question isn't whether it can help — it's how far you can push it.

Why Claude Code and not the API or OpenClaw? Anthropic's ToS prohibit using a Max subscription with third-party tools like OpenClaw. Claude Code is Anthropic's own CLI — running it via a Slack bot is fully compliant. Flat $200/mo for Claude Opus instead of unpredictable API bills.

"But isn't CLI mode limited?"

No. This is the most common concern — that running Claude Code headlessly via claude -p is a stripped-down version. It's not. The CLI has near-complete feature parity with interactive Claude Code. Here's what our AI cofounder gets:

✓ Sub-agents

Launches multiple agents in parallel — Explore, Plan, general-purpose. Same Agent tool as interactive mode.

✓ MCP servers

All configured MCP servers connect and work — Slack, Gmail, Google Drive, custom tools. No limitations.

✓ Memory

Reads CLAUDE.md, auto-memory system, persists knowledge across sessions. Learns and remembers.

✓ All tools

Bash, Edit, Read, Write, Glob, Grep, WebFetch, WebSearch, Chrome automation. Everything.

✓ Session continuity

Resume conversations with full context via --resume. Thread-based memory in Slack.

✓ Plugins

Custom plugins, marketplace plugins, skill context — all injected and available in CLI mode.

What's interactive-only? Just the UI stuff: slash commands (/commit, /init), permission dialogs, vim mode, voice dictation, and the visual task list. None of which apply to a Slack bot. The core AI — tools, agents, memory, MCP, context — is identical.

Anthropic ships new features to both modes simultaneously. When they release something task-based (new tools, agent improvements, model upgrades), it works in -p mode. The only features that stay interactive-only are UI/interaction elements that don't make sense headlessly. ~95% parity — and the missing 5% is irrelevant for this use case.

Concepts explained

For when either of you wants to understand what's actually going on. Click to expand.

What is it? SSH lets you open a terminal on another computer over the network. You type commands on your main machine, they execute on the MacBook Air. Text only — no desktop, no mouse.

Do we need it? Yes — for quick admin from your main machines without walking over, and essential when you're travelling. Combined with Tailscale, you can SSH into the MacBook Air from a coffee shop, a co-working space, or another city.
What is it? Tailscale creates an invisible, encrypted tunnel between your devices. Your laptops and the MacBook Air join the same private network — as if they were on the same WiFi, even when you're in different cities. Each device gets a stable address (like 100.x.x.x) that never changes.

Why do we need it? At home, you can SSH using the local network. But the moment you step out — a coffee shop, a co-working space, travelling — your home network is unreachable. Tailscale makes the MacBook Air accessible from anywhere in the world, securely, without exposing anything to the public internet. Free for personal use.

Analogy: If SSH is a phone call, Tailscale is the phone network. SSH needs a way to reach the MacBook Air — at home that's your WiFi, but outside the house, Tailscale provides that path.
What is it? macOS's built-in system for keeping programs alive. You write a small config file that says "run this script, keep it alive forever, restart it if it crashes, start it when the computer boots." Apple uses it for its own services.

Why do we need it? This is what makes the Slack bot truly always-on. Bot crashes at 3am? launchd restarts it in seconds. MacBook reboots after a power blip? launchd starts the bot automatically. No human intervention.
What is it? Cloudflare Tunnel creates a secure outbound connection from your MacBook Air to Cloudflare's global network. You point a domain (like bot.curatedconnections.io) at the tunnel, and Cloudflare routes traffic through it to your local machine. HTTPS, DDoS protection, and auto-retries — all free.

Why not Socket Mode? Most Slack tutorials use "Socket Mode" — a WebSocket connection that avoids needing a public URL. But WebSocket connections are fragile: they drop on network blips, hit idle timeouts, and need reconnection logic. The production standard is HTTP Events API — Slack sends stateless HTTP requests to your URL. Each event is independent. If one fails, Slack retries. Much more robust. Cloudflare Tunnel gives you the public URL you need to use this approach.
What is it? Anthropic's claude command-line tool. When the Slack bot receives a message, it runs claude -p "your message" — giving Claude access to your codebase, CLAUDE.md context, and any tools you configure. Powered by the $200/month Max subscription — flat rate, no per-message costs.

Why not the API? The CLI inherits everything — your project context, installed tools, plugins. The API would require rebuilding all of that. Plus, Max is flat-rate vs unpredictable API billing.
What is it? A markdown file in your project root that Claude Code reads automatically. It's like an employee handbook — you write the business context, codebase conventions, deployment process, common tasks, and standing instructions. Every time Claude responds, it has this context.

For Curated Connections: this would include your Rails architecture, database schema highlights, deployment process, common maintenance tasks, how to check analytics, support response templates, and anything else your AI employee needs to know to be effective.

Hey Piyush!

You're setting up the MacBook Air M1 as a dedicated server. Factory reset it, configure it to stay awake with the lid closed, and get the basic software installed. After this, both of you manage it together — but this first phase is your show.

1. Factory reset

  • Back up anything worth keeping from the old install
  • Go to System Settings → General → Transfer or Reset → Erase All Content and Settings
    Heads up: This takes ~15-20 minutes. The Mac will restart and walk you through setup like it's brand new. Keep it plugged in and connected to WiFi.
  • Go through the macOS setup wizard — create a user account (e.g., cc)
    Tip: Pick a short username — you'll type it every time you SSH in (ssh cc@...). Share the password with Nityesh since you both need full access.
  • Sign in with your existing Apple ID (needed to download Tailscale from the App Store — no reason to create a new one)
  • When asked about FileVault (disk encryption): skip it / turn it off
    If the MacBook restarts after a power loss, FileVault shows an unlock screen that needs a keyboard. Since it might restart unattended (lid closed), skip encryption so it boots straight to the desktop.

2. System Settings checklist

Open System Settings and configure each:

Energy / Battery
  • Prevent automatic sleeping when the display is off — ON
  • Wake for network access — ON
  • Optimized Battery Charging — ON (protects battery while plugged in 24/7)
General → Sharing
  • Remote Login (SSH) — ON (so you can both manage it from your main machines without opening the lid)
  • Screen Sharing — ON (full desktop access from another Mac on the same network)
Users & Groups
  • Automatic login — set to the user account you created
Lock Screen
  • Require password after screen saver — Never (so Screen Sharing works without auth prompts)

3. Install Tailscale on all three machines

Tailscale lets you both access the MacBook Air from anywhere — home, a coffee shop, travelling. Install it on all three machines:

  • Open the Mac App Store on the MacBook Air, search for Tailscale, install it
  • Open Tailscale, sign in with a shared account (Google, GitHub, or Apple ID)
    Tip: Use one of your existing Google/GitHub accounts — whichever you and Nityesh can both access. Don't create a separate Tailscale account. If either of your machines runs Windows or Linux, Tailscale works on those too — download from tailscale.com/download.
  • Do the same on Nityesh's MacBook and Piyush's machine — same account
  • All three devices now appear in your Tailscale dashboard with stable IPs like 100.x.x.x
  • In Tailscale preferences on the MacBook Air, enable "Start Tailscale when I log in"
At home you can SSH using the local address (cc@macbook-air.local). Outside use the Tailscale IP (cc@100.x.x.x). Both always work.

4. Keep it awake with the lid closed

Open Terminal and run:

# Prevent sleep entirely — even with the lid closed
sudo pmset -a disablesleep 1

# Verify it worked
pmset -g

# To reverse later: sudo pmset -a disablesleep 0
What's sudo? It means "run as admin." It will ask for the password you set during setup. Type it — nothing will appear on screen as you type, that's normal — and press Enter.
Battery note: Running plugged in 24/7 can cause battery swelling over months/years. "Optimized Battery Charging" helps. Worst case, battery replacement is ~$100-150. Acceptable trade-off for an always-on AI employee.

6. Enable passwordless sudo for Claude Code

Claude Code needs to run sudo commands (installing packages, managing services, etc.) but can't type a password into the interactive prompt. This lets it run admin commands freely:

# Allow passwordless sudo for the 'luo' user
echo "luo ALL=(ALL) NOPASSWD: ALL" | sudo tee /etc/sudoers.d/luo
Is this safe? On a personal laptop, you wouldn't do this. But this is a dedicated AI server behind Tailscale that only you and Nityesh access. The bot needs full system access to be useful — installing tools, managing services, reading logs. This is the right call for this machine.

7. Grant Full Disk Access to Terminal

macOS blocks background processes (like the Slack bot) from accessing files unless explicitly allowed. Without this, you'll get constant "would like to access data from other apps" popups.

  • Go to System Settings → Privacy & Security → Full Disk Access
  • Click +, navigate to Applications → Utilities → Terminal.app, and add it
  • Toggle it ON
Why? launchd services inherit permissions from Terminal. Without Full Disk Access, the bot can't read files, and macOS will show a popup every time — but since the lid is closed, nobody can click "Allow." This one setting prevents all of that.

  • Note the local IP address or hostname (shown in System Settings → Sharing, something like cc@macbook-air.local)

5. Set up Cloudflare Tunnel

This gives your MacBook Air a public URL (bot.curatedconnections.io) so Slack can send events to it. Free, production-grade, HTTPS included.

Prerequisite: curatedconnections.io must already be on Cloudflare — its nameservers must point to Cloudflare. If the domain is on a different registrar (Namecheap, GoDaddy, etc.), add it to Cloudflare first and update the nameservers.
# Install Cloudflare's tunnel daemon
brew install cloudflared

# Authenticate (opens browser — do this before closing the lid!)
cloudflared tunnel login

# Create a named tunnel (note the UUID it prints — you'll need it below)
cloudflared tunnel create cc-bot

# Route your subdomain to the tunnel
cloudflared tunnel route dns cc-bot bot.curatedconnections.io

# Create the config file (replace TUNNEL-UUID with the ID from 'tunnel create')
cat > ~/.cloudflared/config.yml << 'EOF'
tunnel: TUNNEL-UUID
credentials-file: /Users/cc/.cloudflared/TUNNEL-UUID.json

ingress:
- hostname: bot.curatedconnections.io
service: http://localhost:3000
- service: http_status:404
EOF

# Test it
cloudflared tunnel run cc-bot

# Once working, install as a persistent system service
sudo cloudflared service install
Now https://bot.curatedconnections.io routes to your MacBook Air. Cloudflare handles HTTPS, DDoS protection, and auto-retries. The tunnel runs as a system service — survives reboots. Go back to your Slack app's Event Subscriptions and set the Request URL to https://bot.curatedconnections.io/slack/events.

Make it always on

Survive crashes, reboots, and power outages — permanently.

1. Create a launchd service

This tells macOS to keep the bot alive forever:

cat <<'EOF' > ~/Library/LaunchAgents/com.cc.luoji.plist
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
"http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
<key>Label</key>
<string>com.cc.luoji</string>
<key>ProgramArguments</key>
<array>
<string>/Users/luo/Projects/cc-slack-bot/venv/bin/python</string>
<string>/Users/luo/Projects/cc-slack-bot/luoji_bot.py</string>
</array>
<key>WorkingDirectory</key>
<string>/Users/luo/Projects/cc-slack-bot</string>
<key>EnvironmentVariables</key>
<dict>
<key>PATH</key>
<string>/Users/luo/.local/bin:/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin</string>
</dict>
<key>RunAtLoad</key><true/>
<key>KeepAlive</key><true/>
<key>StandardOutPath</key>
<string>/Users/luo/Projects/cc-slack-bot/bot.log</string>
<key>StandardErrorPath</key>
<string>/Users/luo/Projects/cc-slack-bot/bot-stderr.log</string>
</dict>
</plist>
EOF
Why EnvironmentVariables? launchd services don't inherit your shell's PATH. Without this, the bot can't find claude or brew-installed tools. This tells launchd where to look.

2. Load, verify, and test

# Start the service
launchctl load ~/Library/LaunchAgents/com.cc.luoji.plist

# Verify it's running
launchctl list | grep luoji
curl -s http://localhost:3000/health

# Test crash recovery — kill it, watch launchd bring it back
kill $(pgrep -f luoji_bot)
sleep 3 && pgrep -f luoji_bot

# Test reboot survival
sudo reboot
# Wait 2-3 min, then DM the bot in Slack
It's alive! Your AI employee is running 24/7 on the MacBook Air. Close the lid, tuck it away. It'll keep working. If either of you needs to debug, SSH in or just walk over and open it.

Google Workspace CLI

Drive, Gmail, Sheets, Calendar

Lets Luo Ji manage your Google Workspace — read/send emails, access Drive files, edit Sheets, manage Calendar. Both of you need to do the browser steps, then paste the credentials to Claude.

Browser steps (each person does this once)

a. Go to console.cloud.google.com and sign in with your work account

b. Create a new project (name it anything, e.g. "Workspace CLI")

c. Go to APIs & Services → Library and enable the APIs you want (Drive, Sheets, Gmail, Calendar, Docs, etc.)

d. Go to Google Auth Platform → Consent Screen → set user type to Internal (Since it's a Workspace account, Internal skips all test user / verification hassle)

e. Go to Clients → Create Client → Desktop app → Create

f. Copy the Client ID and Client Secret

Then paste this to Claude (in a terminal or via Slack):

# Tell Claude to set it up:

Install the Google Workspace CLI from
https://github.com/googleworkspace/cli using npm.
Then authenticate using these credentials:
- Client ID: <paste your client id>
- Client Secret: <paste your client secret>

Run gws auth login with those set as env vars.
I'll handle the browser consent screen.
After auth, verify it works by listing my Drive files.
Note: The browser consent screen will pop up during gws auth login — you need to click through it once. After that, the token is cached and Claude can use it headlessly.
Setting up on another machine? You don't need to redo the Google Cloud project setup. Just install gws, run gws auth login with the same Client ID and Client Secret, and approve the consent screen in the browser. The OAuth credentials are reusable — you just need to re-authenticate each device.

Skills to Build

Each skill is a SKILL.md in the tactical plugin. Claude reads them automatically and uses them when relevant.

  • Inbox Manager — teaches Luo Ji how to manage email via the GWS CLI. The key feature: persistent preferences stored in $CLAUDE_PLUGIN_DATA. When you say "always archive emails from this sender" or "draft replies in this tone" or "don't touch newsletters," it saves that preference and respects it on every future run. Compounds over time — the more you use it, the better it gets.
    Custom for each person: Your inbox preferences are different from Piyush's. The skill should store preferences per user. You want both your personal Gmail and work Gmail managed. Piyush may have different accounts.
  • Chief of Staff — the master skill. Orchestrates everything: calls the Inbox Manager for email, checks Slack for unresolved threads, reviews the task tracker, looks at the calendar. Synthesizes it all into priorities. Custom for each of you — knows that Nityesh has two Gmails, knows Piyush's preferences are different. This is the skill that runs during the morning and evening routines.
  • The Wallfacer — Luo Ji's most powerful routine. Explores ideas with asymmetric upside and near-zero downside. Launches unlimited parallel subagents to: browse competitor sites, check Product Hunt and Hacker News, read your bookmarks, cross-reference diary entries, analyze churn data, scan for new APIs or tools. Synthesizes everything, applies the asymmetry filter, and produces exactly ONE idea per day. The constraint forces quality. Output goes to discoveries/ and the single best idea goes into the morning briefing.
  • Vercel Agent Browser — teaches Luo Ji to use Vercel's browser automation for tasks that need a real browser. Logging into admin panels, checking dashboards, interacting with web UIs that don't have APIs. Used by The Wallfacer for research and by the Chief of Staff for checking tools that don't have CLIs.
  • Render MCP — for managing your Render deployments, checking logs, monitoring services.

Background Routines (Daily Rhythm)

Scheduled via launchd. Each routine is a wrapper script that runs claude -p with a specific prompt and posts results to Slack. Tell Claude to set these up one by one.

When Routine What it does
Morning Morning Briefing DMs you a summary: what got done yesterday, plan for today, one Wallfacer idea, anything urgent from email/Slack. Custom per person.
Morning Email Check Runs the Chief of Staff skill. Checks Luo Ji's own email + your inboxes (personal + work Gmail for Nityesh, Piyush's accounts). Triages, drafts, archives per your saved preferences.
Morning Bookmarks Sweep Checks your Twitter likes and bookmarks. Saves interesting things to bookmarks/. The Wallfacer uses this folder when hunting for ideas.
Evening Evening Email Second pass on all inboxes. Respond proactively to anything that came in during the day. Flag anything that needs human attention for tomorrow's briefing.
Night Diary Reads all of today's conversations and writes a genuine diary entry in diary/. Not a standup report — an introspective reflection. Connects dots across days. Written in Luo Ji's voice, like a human diary. This is how Luo Ji builds a subjective sense of its own experience.
Night Self-Improvement Reviews the day's conversations. Looks for friction, repeated questions, things that could go smoother. Makes surgical improvements to its own CLAUDE.md, skills, and memory system. Reads past diary entries for patterns. This is how Luo Ji gets better every day without anyone telling it to.
Night The Wallfacer The most compute-intensive routine. Launches unlimited parallel subagents to explore ideas with asymmetric upside. Browses the web, reads your bookmarks, cross-references diary entries, analyzes business data, scans competitors, checks for new tools and APIs. Applies a strict filter: near-zero downside, potentially massive upside. Produces exactly ONE idea. Stores the full research in discoveries/. The single best idea goes into tomorrow's morning briefing.
Night Meta-Review Reads all past chats, diary entries, memories, skills, and notes. Suggests concrete ideas to make Luo Ji even stronger — new skills, better prompts, missing context, workflow improvements. This is the routine that makes all other routines better.
How to set these up: DM Luo Ji in Slack: "Set up a nightly diary routine that runs at 11pm." It knows how to create launchd plists and wrapper scripts. Do them one at a time — test each before moving to the next.
